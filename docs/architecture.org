#+TITLE: Autolith Architecture
#+STARTUP: content

* Intent

Autolith is designed to understand and modify itself while it is running. The
architecture therefore optimizes for semantic boundaries that are easy to
inspect, document, replace, and test independently.

Every application definition lives in the single =autolith= package. Files
separate responsibilities; packages do not.

The central objects communicate through generic functions. Implementations
may be replaced without teaching the agent loop about HTTP, terminal escape
sequences, source editing, or subprocess protocols.

* Responsibilities

** =configuration.lisp=

Defines immutable runtime choices and paths. Configuration contains credential
locations, never credential values.

** =sexp-store= and =readable-state.lisp=

The pinned =sexp-store= dependency owns strict one-form reading, atomic private
S-expression snapshots, append-only readable logs, and incomplete-tail repair.
Readers disable =*read-eval*=, snapshot consumers reject trailing forms, and no
consumer sees a half-published replacement. =readable-state.lisp= retains only
the property-list key predicate used by Autolith record validators.

** =authentication.lisp=

Defines credential sources and OAuth refresh. Credentials enter memory only
inside the dynamic scope of a provider request. The Codex credential file is
an optional one-time, access-token-only import, not a runtime dependency on
Codex. Renewable credentials come only from Autolith's own device flow.

** =device-authentication.lisp=

Owns Autolith's browser-assisted device authorization, polling, PKCE exchange, and
publication into Autolith's private credential store.

** =conversation.lisp=

Owns append-only portable records and their projection into provider input.
Fresh conversations remain in memory until the first durable record atomically
publishes the header and that record together. Empty sessions therefore create
no conversation file. It persists each user message before network work begins
and treats completed provider output items as authoritative history. The header
records the initial model and reasoning effort; later selection changes append
configuration records so resuming restores the latest choices. Loading records
whether a crash interrupted the final form, and the next append asks
=sexp-store= to remove that tail atomically before publishing new history.

** =memories.lisp= and =memory-tools.lisp=

Own the append-only portable memory log, replacement replay, recall tombstones,
workspace and global selection, deterministic lexical ranking, and the
=memory.*= tool surface. =memory-context.lisp= contributes a bounded set of
metadata and short excerpts when relevant memories match the latest user input.
That request-local evidence is quoted as untrusted JSON data and never enters
conversation history; exact bodies remain behind search and read tools. A
damaged memory log yields a bounded contributor failure instead of breaking the
provider path. An incomplete final form is ignored and atomically removed when
the next complete record is appended.

The design was informed by OpenAI Codex commit
=5bed6447998c754d154dbd796517310b8f04d4ce=. That implementation uses SQLite
for background extraction jobs and usage ranking, then consolidates selected
results into a Markdown tree whose bounded =memory_summary.md= enters developer
instructions. Autolith keeps the same useful separation between small ranked
recall evidence and deeper read tools, but its single-agent runtime does not
need a job database, background extraction model calls, Git-backed consolidation
workspace, or generated Markdown hierarchy.

** =agendas.lisp= and =agenda-tools.lisp=

Own the atomic versioned workspace agenda file, legacy migration, stable item
identifiers, memory attachments, repository transport, and the =agenda.*= tool
surface. An item stores only stable memory identifiers, so replacing the
append-only memory record changes the value behind every attachment without
rewriting agenda state. Transport preserves attachments and unions them when
otherwise duplicate items merge.

** =provider.lisp=

Owns the replaceable model-provider protocol, Responses Lite request encoding,
HTTP transport, and SSE decoding. It emits semantic stream events and has no
terminal rendering logic.

** =tools.lisp=

Defines tool metadata, namespaces, validation, dispatch, and output. Every
tool call receives a matching result, including malformed and failed calls.

** =default-tools.lisp=

Composes the default registry from one focused registration function per
namespace. This keeps a schema or tool-set change local instead of requiring a
replacement of the complete registry definition in the live image.

** =search-tools.lisp= and =search-worker.lisp=

=search-tools.lisp= owns the pinned native-library identity, tool classes, and
argument policy. =search-worker.lisp= adapts Autolith configuration and tool
results to =clifff=. The pinned =clifff= dependency owns the CFFI binding,
workspace index, readable result data, bounded presentation, supervised helper
protocol, process retry, and rebuildable database reset. One registry shares
one worker across fuzzy path, glob, content, and multi-pattern search; the
helper process owns the single native fff engine. The Rust library is built from
pinned fff commit
=44a5b259570730a4236ecbf06673d43ef7b2263e= and published privately under the
Autolith data root. It is neither a command subprocess nor a program installed
on the user's =PATH=. This integration was reviewed against that exact upstream
commit's generated C header and Rust implementation.

The helper owns all native watcher threads, caches, database handles, and fff
pointers. A dead process or failed exchange closes it, discards only rebuildable
ranking state, and retries once in a fresh process. Checkpoint entry and normal
application exit close the helper, so the active image never saves or inherits
a native fff pointer. A resumed application starts a fresh helper on its next
search.

** =lisp-worker.lisp=

Owns named, persistent SBCL subprocesses, their readable S-expression protocol,
matching implementation-source lookup, and forked worker-core saving. No
=lisp.*= operation shares heap state with another REPL or the active image.

** =lisp-images.lisp=

Owns immutable worker-image manifests, compatibility checks, parentage, durable
notes, atomic core publication, and the bounded inventory placed in every model
request. The pristine image is a reserved virtual root.

** =self-tools.lisp=

Owns active-image inspection, exploratory mutation, definition identity, and
form-aware source primitives. Package-aware definition installation can
instrument Lisp-level SBCL packages while restoring package locks afterward.

** =durable-mutations.lisp=

Owns the checked one-definition transaction and validated durable journal replay.
Durable mutation objects carry explicit phases and reconcile against published
private image commits across the narrow post-publication crash window.

** =image-commits.lisp=

Owns stable exploratory mutation identities, branch lineages, effective
=self.diff= state, =self.commit=, private commit manifests, complete Lisp replay
scripts, and the clean-process replay probe. Each commit is a full effective
snapshot under the data root. The snapshot is replay-probed before selection and
then committed to the separate mutation-history Git repository under the state
root. An atomic pointer binds both identities and selects normal startup state.
Startup restores deleted canonical manifest and replay files from the exact Git
object. Retained cores preserve their checkpointed identities and begin a new
lineage on reconnect, so rollback cannot absorb mutations written later by
another branch. Legacy overlays are loaded at the base and imported into the
first private commit.

** =self-status.lisp=, =self-discard.lisp=, and =self-exercise.lisp=

Expose concise mutation and recovery state, exact reversible discard of one
effective exploratory mutation, and focused append-only exercise evidence.
These operations use the mutation identity, lineage, and replay primitives
without changing the tracked source repository.

** =terminal.lisp=

Owns a restrained primary-screen interface. Finalized transcript output is
written once and left in normal terminal scrollback. Autolith never enters the
alternate screen and never clears scrollback. Only the bounded live region,
holding transient reasoning previews, unfinished status, the full-width multiline
prompt, and any command completion suggestions, may be repainted. Reasoning
previews enter scrollback only when finalized. Active status includes phase
elapsed time and identifies how long no provider progress has arrived after a
bounded idle interval. Clinedi owns Unicode-aware editor state, input decoding,
screen geometry, candidate navigation and arrangement,
and the scrollback-safe live-region mechanics. Autolith owns candidate data,
command filtering, semantic styling, terminal mode, and application event
policy. Kernel TTY rows and columns take precedence over environment hints.
Long drafts use a cursor-following viewport capped below the terminal height,
and each replacement is emitted as one hidden-cursor update.
The startup identity uses a
capability-checked xterm-256 green gradient with bold basic-green fallback,
while a process started by recovery uses a red gradient with bold basic-red
fallback. One leading blank row and a divider separate its identity from model
and workspace metadata. Internal conversation identifiers are omitted. Other
semantic styles resolve to basic terminal palette colors and emphasis. Untrusted
text is sanitized before trusted styling controls are added around it. Ctrl-C at
an empty prompt exits and leaves a paste-ready resume command in scrollback only
when the active conversation is durable.

** =responsive-input.lisp=

Owns the ephemeral terminal-reader thread and FIFO submitted-message queue.
Model turns and commands remain on the application main thread, while the reader
applies editor events through the terminal UI lock. This keeps drafts editable
during provider streaming and tool execution without allowing concurrent model
turns. Modal pickers temporarily join the reader and take terminal ownership.
Checkpoint creation uses the same nested quiescence boundary, so its process
fork still occurs with exactly one live Lisp thread. The reader restarts after
the modal or checkpoint operation. This controller is lexical to one
application run and is never captured in a retained generation.

** =agent.lisp=

Coordinates one turn at a time: persist user input, stream a response, append
completed output items, execute tools, append their results, and continue
until the provider completes without outstanding calls. A long turn receives a
model-visible warning at provider step 48. Step 64 advertises no tools and asks
for a text-only completion. A separate 256-call ceiling bounds unusually large
tool batches. Exhausting either guard is expected turn control, not a reason to
boot recovery.

** =generations.lisp=

Owns compatible manifests and the Linux SBCL coordinator-and-saver checkpoint
backend. An unpublished core must boot and identify its exact generation and
tracked source commit together with its private image commit and mutation-history
Git commit before its core, manifest, and selection become visible. Checkpoint
preparation commits pending reconstructible mutations and writes a complete
=reconstruct.lisp= beside every new core. Manifest version 3 records all three
identities; versions 1 and 2 remain readable.

** =main.lisp=

Roots logical application state for checkpoints, reconnects ephemeral resources,
projects durable records into the terminal, and wires interactive commands.

** =active-image.lisp=

Builds and validates the credential-free preloaded baseline image used for fast
normal startup. The embedded build record hashes every compiled source input
from Autolith, includes the dependency lock, and records the exact runtime. Its
private probe refuses stale source or an incompatible runtime before the stable
launcher can use the core.

** =bin/= and =recovery/=

Contain the stable launcher boundary, pinned-runtime adapter, and minimal
recovery runtime. =script/bootstrap= locates or installs the pinned SBCL beneath the
data root without changing the system Lisp. It also installs the independently
hash-verified matching SBCL source as a read-only reference. It builds the
recovery runtime into a read-only pristine core and builds the separate
preloaded active image outside the repository. Both builders record exact
source provenance and boot-probe each temporary image. The launcher repeats the
versioned probes before use. An interactive missing-image path offers to run the
bootstrap in place, while declining or a non-interactive launch uses source.
=--from-source= bypasses the image probe and prompt entirely.
Recovery reports
sanitized crash and mutation context, creates private exact-commit checkouts,
and boots retained cores without loading damaged active source. A failed
retained core restores terminal state, refreshes crash context, and falls through
to another compatible generation. When no retained generation can boot, recovery
starts a clean checkout of the current committed source. Rollback selection uses
a dedicated process status to enter this path immediately.

* Durable mutation

A durable change follows one visible transaction across otherwise independent
components:

1. Append the proposed mutation to the journal.
2. Compile and install it in the active image.
3. Run the relevant checks.
4. Publish a complete private image-commit replay script and manifest.
5. Commit the complete snapshot to the private mutation-history Git repository.
6. Atomically select the image and Git commit identities for normal startup.
7. Append the durable result to the journal.

Exploratory mutation stops after installation and journal publication until
=self.commit= consumes it. Tracked source remains authoritative for the base
system, and the selected private script reconstructs durable user-specific
changes without patching that source.

* Credentials and saved images

Provider instances retain credential-source pathnames but not access or
refresh tokens. Checkpoint code must run outside the dynamic request scope and
must reject attempts to save while credentials are dynamically bound. Tokens
never enter conversations, memories, journals, logs, or manifests. Worker
images contain only isolated REPL heaps and never participate in active-image
startup or recovery. Their manifests and notes live under the data root; the
pinned runtime executable and pristine recovery artifacts remain separate.
