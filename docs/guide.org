#+SUBTITLE: Setup, operation, tools, persistence, and recovery

* What works

Autolith is an initial dogfoodable Common Lisp agent for Linux and SBCL. It:

- talks directly to the ChatGPT Codex subscription Responses service;
- defaults to =gpt-5.6-sol= with user-visible reasoning effort =ultra=;
- owns its system prompt, conversation state, tool loop, and authentication;
- recalls workspace and global persistent memories through dedicated tools;
- maintains short workspace agendas and recalls them in full;
- never invokes or depends on the Codex executable;
- keeps finalized output in normal terminal scrollback;
- runs =lisp.*= operations in named, heap-isolated SBCL REPLs;
- saves modified worker heaps as immutable images with durable notes;
- inspects the exact source matching its pinned SBCL runtime;
- exposes documented =self.*= inspection and mutation operations;
- privately commits reconstructible live mutations without committing to Git;
- saves live SBCL generations without stopping the active parent;
- keeps a separate pristine recovery core;
- starts from a validated preloaded active image when available; and
- can boot a retained generation against its exact Git revision with fallback.

The implementation follows [[file:autolith-minimal-technical-spec.org][the technical specification]]. The object and
file boundaries are summarized in [[file:architecture.org][the architecture note]].

* Requirements and setup

The supported target is Linux x86-64 with Git, OpenSSL, and Bubblewrap.

** Nix package

Nix is the recommended installation path because it supplies the complete
runtime and native dependencies without borrowing them from the host.

#+BEGIN_SRC sh
nix run github:luciusmagn/autolith
#+END_SRC

The first launch creates private recovery and active images without network
access.

** Binary release

The conventional installer downloads a versioned release archive and its
SHA-256 record, verifies the archive, installs it atomically below
=${XDG_DATA_HOME:-~/.local/share}/autolith/installation/=, and links =autolith=
into =~/.local/bin/=.

#+BEGIN_SRC sh
curl -fsSL https://sh.lambda-symbolics.com/autolith | sh
#+END_SRC

Piping a remote script into a shell is intrinsically difficult to audit. Read
the script first or use Nix when that tradeoff is unwelcome. A binary release
contains pinned SBCL 2.6.4, its matching source, the locked Lisp dependency
tree, fff, and the private command sandbox helper. It does not require a source
checkout, Quicklisp, Cargo, or a compiler. Its first launch automatically builds
machine-specific pristine recovery and active images from those bundled inputs.

Autolith uses the installation layout, not a user-controlled environment marker
alone, to distinguish source, Nix, and selected binary-release starts. Source
checkouts do not contact the release service. Nix starts may show a cached
advisory to update the flake or profile, but Autolith never changes a Nix
installation itself.

For a selected binary release, Autolith reads only the previous cached result
during startup. A newer nondismissed release appears below the banner and offers
=Not now=, =Update now=, and =Skip this version=. Not now affects only that
startup. Skipping suppresses the exact tag, so a later release appears normally.
After the interface is ready, a bounded background request refreshes the private
readable cache when its last attempt is at least 20 hours old. Failure is
nonfatal, preserves the last valid result, and never triggers an installation.

Update now first restores the terminal and exits the active application through
a dedicated non-recovery status. The stable outer release launcher then checks
the newest tag again, invokes the installer bundled in the trusted selected
release, verifies the downloaded archive against its SHA-256 record, atomically
selects it, and restarts with the original arguments. The updater derives and
preserves a custom installation root and does not replace the user's command
link. =autolith --update= requests the same verified launcher path explicitly.
Running the installer again is also an update and keeps older versioned releases
available under the installation root.

Tagged Linux archives are built and checked by the installer host, then served
by the repository's Common Lisp release server. GitHub CI publishes only a
metadata compatibility record after the host checksum becomes available. See
[[file:release-service.org][the release service note]] for the build, publication, and deployment flow.

** Source checkout

Source development additionally needs Quicklisp, =curl=, GNU =tar=,
=sha256sum=, Rust with Cargo, a C compiler, CMake, and =pkg-config=. The tracked
=sbcl.version= pins SBCL 2.6.4. The setup command uses a matching
=AUTOLITH_SBCL= when explicitly configured, locates a matching system runtime,
or downloads the official Linux x86-64 binary and checks its pinned SHA-256
identity before installing it under the Autolith data root. Setup also downloads
the official matching SBCL source archive, verifies its separately pinned
SHA-256 identity, and publishes the extracted tree without write bits beside
the runtime. The launcher and development commands remember that selection.

Dependencies are pinned by =qlfile= and =qlfile.lock=. With Quicklisp already
installed at =~/quicklisp=, run:

#+BEGIN_SRC sh
./script/bootstrap
./script/check
#+END_SRC

=script/bootstrap= installs the pinned runtime when needed, installs Qlot through
Quicklisp, materializes the locked project dependencies under the ignored
=.qlot/= directory, builds the private command sandbox helper and fff library,
builds a small pristine recovery core under
=${XDG_DATA_HOME:-~/.local/share}/autolith/recovery/=, and builds a preloaded
active image under the adjacent =active/= directory. Both cores and their
manifests are installed read-only. Matching implementation source lives under
=${XDG_DATA_HOME:-~/.local/share}/autolith/runtimes/2.6.4/source/= and is used as
a read-only inspection reference, never as a live patch target.

The launcher probes the preloaded image against the current SBCL, operating
system, architecture, and content hash of every Autolith source input plus the
dependency lock. A valid image avoids dependency and application loading on each
launch. When it is missing or stale during an interactive launch, Autolith asks
whether to run =script/bootstrap= immediately. Accepting rebuilds and starts the
new validated image; declining loads from source. Non-interactive launches fall
back directly to source. Pass =--from-source= to deliberately skip image
validation, the bootstrap prompt, and missing-image notices. The source loader
and check script prefer the project-local dependency environment, then fall back
to the user's Quicklisp.

On systems where OpenSSL is outside the dynamic loader's normal search path,
make its library directory visible to CFFI. Autolith automatically adds
=~/.guix-profile/lib/= when it exists.

To make the development checkout available in =PATH=:

#+BEGIN_SRC sh
mkdir -p ~/.local/bin
ln -sfn "$PWD/bin/autolith" ~/.local/bin/autolith
#+END_SRC

* Authentication

Autolith stores its own credentials in
=${XDG_STATE_HOME:-~/.local/state}/autolith/auth.sexp= with mode 0600.

For the initial transition, Autolith may read =~/.codex/auth.json= once as a read-only
bootstrap source when its private store is absent. It copies only the bounded
access token and account ID into Autolith state. It never imports or spends Codex's
refresh token, never reads Codex again while its private store exists, never
writes that file, and never launches Codex. The imported access token cannot be
renewed. To perform Autolith's proper device flow, run:

#+BEGIN_SRC sh
autolith --auth
#+END_SRC

The same flow is available as =/auth= inside the application. Autolith prints the
public verification URL and one-time code, tries to open a browser, polls the
device endpoint, performs the PKCE exchange, and writes only its private store.

* Using Autolith

Start a new conversation with:

#+BEGIN_SRC sh
autolith
#+END_SRC

Use =autolith --immutable= when the active image should expose only
=self.inspect=, =self.source=, =self.status=, =self.diff=, and
=self.generations=. The flag removes evaluation, mutation, persistence,
checkpoint, and rollback tools from the provider schema. It does not turn
filesystem, shell, or worker operations into a security sandbox.

The interface stays on the primary screen. Completed transcript rows are emitted
once and remain in ordinary terminal scrollback. Only the bounded live region,
holding unfinished trace previews, status, the full-width prompt, and command
suggestions, is repainted. Clinedi provides Unicode-aware editing and a cursor-following
multiline viewport. Input wraps at the actual TTY width, explicit input lines
are preserved, and the live region never owns the terminal's complete height.
Each edit replaces that region in one hidden-cursor terminal write. Model and
pasted text are sanitized before they reach the terminal. The startup =AL= mark
uses a light-green xterm-256 row gradient when available and solid bold green
otherwise. A process started by recovery uses the corresponding light-red
gradient and solid bold red fallback. Its version heading and divider sit
beside the mark above the model and workspace; internal conversation identifiers
stay out of the header. Other styles use the basic terminal palette and
emphasis. cl-colorist owns terminal color capability detection and ANSI style
encoding, while Autolith retains its semantic palette. Set =NO_COLOR= to disable
styling entirely.

Tool details, Markdown tables, and interactive picker descriptions use the same
terminal-cell-aware column allocator. Long labels and values shrink or wrap
within the current width instead of relying on tabs or fixed character columns.

Enter submits the complete input. Shift-Enter, Ctrl-Enter, and Alt-Enter insert
a newline. Autolith enables CSI-u and modifyOtherKeys reporting while it owns the
terminal and restores ordinary keyboard reporting on exit. Up and Down preserve
the preferred terminal cell column across explicit and wrapped input rows,
falling back to history only at
the first or last visual row.

Paste or drag one or more local PNG, JPEG, GIF, or WebP pathnames into the
composer to attach images. Autolith replaces each pathname with =[Image #N]=,
keeps the attachment through multiline editing, history, queuing, and steering,
and removes it if its label is deleted. Surviving labels are renumbered before
submission. Start with repeatable =-i FILE= or =--image FILE= options to preload
images into the composer; comma-separated values are also accepted. Images are
validated and resized in Common Lisp to the current Codex high-detail limits.
Conversation logs retain only portable descriptors, while private binary
artifacts live under the Autolith data root and provider requests reconstruct
their base64 data URLs.

The prompt remains editable while Autolith streams a response or runs tools.
Enter steers the active turn at its next tool boundary. Tab queues the draft as
a later follow-up; the live region shows bounded previews of steering and queued
messages. With an empty draft, Tab recalls the newest follow-up for editing.
When Autolith is idle, Tab submits like Enter. On styled terminals, active
status uses a neutral full-width background and colors the current model,
reasoning effort, and enclosing Git worktree branch with the basic palette.
Branch discovery occurs when the activity phase changes, not during the
four-times-per-second spinner repaint. Active status includes elapsed time;
after 30 seconds without provider progress it also shows the no-update duration.
A draft that has not been submitted survives streaming updates unchanged. Slash
commands that need modal terminal input are held in the editor until the current
response finishes; nonmodal commands such as =/compact= retain responsive input
while they run. Ctrl-C at an empty prompt cancels the active turn, discards
queued messages, and prints the exact resume command for a durable conversation.
If clean cancellation cannot finish, another Ctrl-C restores the terminal and
forces Autolith to exit.

A fresh conversation stays in memory until its first durable record. Starting
Autolith and leaving without a message or other saved state creates no
conversation file and adds nothing to =/resume= or =/conversations=.

Startup and resume print at most the newest 500 transcript entries visible under
the current presentation settings. =/history= appends the preceding page of at
most 500 entries in chronological order. Repeated calls continue paging
backward. Older pages are labeled because terminal scrollback is append-only
and they therefore appear below the newer transcript. This display bound does
not remove anything from durable conversation storage, provider context, or
compaction input.

=Ctrl-C= clears a nonempty input draft. At an empty prompt it exits Autolith;
when the conversation is durable, shutdown prints the exact
=autolith resume ID= command that reopens it. Empty conversations produce no
resume instruction. Pressing Ctrl-C again while shutdown is pending forces exit
with status 130 and repeats the durable resume command before leaving.

Conversation identifiers are short, case-sensitive Bitcoin Base58 values such
as =K-8vQ2mp=. The hyphen after the first character is visual, so both
=K-8vQ2mp= and =K8vQ2mp= resume the same conversation. On first startup after
upgrading from UUID identifiers, Autolith migrates saved conversations and
their durable references transactionally. Old resume commands remain aliases.

Long agent turns are allowed to continue well beyond eight tool batches. Autolith
continues provider and tool rounds until the model finishes, the user interrupts,
or an unrecoverable provider or agent error occurs. Context compaction happens
within the active turn and resumes work without imposing a provider-step or
tool-call ceiling.

Typing =/= opens command suggestions under the prompt. Arrow keys move the
selection, Tab cycles and previews options, and Enter runs the selected command,
first filling in the command name when it still needs an argument. Ordinary
input keeps the previewed option and continues editing; Escape restores the
prefix from before selection. The commands are:

#+BEGIN_EXAMPLE
/help           show this reference
/new            start a new conversation
/resume         pick a saved conversation to resume
/conversations  list saved conversations
/history        load the preceding 500 transcript entries
/cwd PATH       change the active workspace
/auth           authenticate Autolith with ChatGPT
/model          pick the 5.6 model and reasoning effort
/effort         pick the reasoning effort
/trace on|off   show visible reasoning summaries
/permissions    choose command access for this session
/later INPUT    run input after rate limits reset
/goal OBJECTIVE set or view the session goal
/agenda         show workspace agenda entries
/checkpoint     save a retained live generation
/generations    list retained generations
/rollback       start a retained generation through recovery
/status         show usage and rate limits
/compact on|off hide routine tool results; no argument summarizes context
/quit           leave Autolith
#+END_EXAMPLE

=/resume=, =/rollback=, =/model=, and =/effort= open a vertical interactive
picker; each also accepts an explicit argument. Tab cycles the picker alongside
Up and Down navigation. The resume picker lists conversations from the current
directory first and other sessions underneath. Each section is ordered by newest
activity first. Generations are listed newest first.
From the shell, =autolith resume= opens the same picker and may be followed by
an explicit conversation identifier.
Conversations remember the directory they were started in, and the picker
shows it.

Command-line resume may occasionally offer to create =AUTOLITH.org= at the
current project root. It does this only after resuming a conversation whose
recorded activity spans at least an hour, or after the project accumulates two
such conversations. For legacy records without usable times, twelve user turns
is the conservative fallback. =not-now= defers the offer for five days and
=never= suppresses it permanently for that canonical repository or workspace
path. The in-session =/resume= command never initiates the offer.

=/cwd PATH= changes the workspace for the current Autolith process. Relative
paths resolve from the current workspace, =~/= expands to the home directory,
and spaces need no quoting because the complete command remainder is the path.
=/cwd= alone prints the current workspace. A successful change updates subsequent
system prompts and inherited =AGENTS.md= instructions, filesystem and shell tools,
search, workspace memories, workspace agendas, and existing or future Lisp REPLs
without discarding their heaps. It does not rewrite the active conversation's
recorded origin.

=/goal OBJECTIVE= sets a session goal in the spirit of Codex goals. The goal
rides along every provider request as transient context, and after each of
your messages Autolith automatically continues working toward it for up to eight
extra turns before pausing. The model marks genuine completion with a literal
=[GOAL-COMPLETE]= token. =/goal= alone shows the state, =/goal pause= and
=/goal resume= control automatic continuation, and =/goal clear= removes the
goal. Goals persist with their conversation across =/resume=.

=/model= switches between the 5.6 family models (sol, luna, terra), then
prompts for the reasoning effort to use. =/effort= changes only the effort.
The available efforts are =none=, =low=, =medium=, =high=, =xhigh=, =max=,
and =ultra=. Autolith sends the user-visible =ultra= choice as provider effort
=max=.
Both choices become the global defaults for later launches. =AUTOLITH_MODEL=
and =AUTOLITH_REASONING_EFFORT=
override the saved defaults when set at startup. Each conversation also records
its latest model and effort, so =/resume= restores the choices used there.

=/trace on= shows each provider reasoning summary as a bounded, Markdown-styled
live trace above the status and prompt, then finalizes it once in the transcript.
=/trace off= stops requesting and persisting new summaries. The choice is saved
globally in =${XDG_STATE_HOME:-~/.local/state}/autolith/preferences.sexp= and
survives clean restarts. Autolith never displays raw reasoning text or encrypted
reasoning payloads; these are model-provided summaries, not hidden chain-of-thought.

** Child agents and jobs

Use =task.agents= to inspect the child roles the caller may select and any
rejected role files. It reports each effective role's description, source,
pathname, model and effort policy, tool and spawn policy, and output-contract
presence. Project roles under =.autolith/agents/= override user roles under
=${XDG_CONFIG_HOME:-~/.config}/autolith/agents/=, which override bundled roles.
A malformed higher-precedence role is reported and never silently replaced by a
lower-precedence role with the same name.

Each user or project role is one bounded UTF-8 =*.sexp= file containing exactly
one top-level Common Lisp property list and nothing after it. Autolith reads the
form with =*read-eval*= bound to =nil= and a fresh =(copy-readtable nil)=. The
only keys are =:name=, =:description=, =:instructions=, =:tools=, =:spawns=,
=:models=, =:reasoning-effort=,
=:output=, and =:blocking-p=, each at most once. The first three are required
nonempty strings. For example:

#+BEGIN_SRC lisp
(:name "reviewer"
 :description "Review a change for correctness and regressions."
 :instructions "Report concrete, evidence-backed findings."
 :tools ("fs.read" "search.*")
 :spawns ("scout")
 :models ("@parent")
 :reasoning-effort :high
 :output (:type :object
          :properties (("answer" (:type :string))
                       ("findings" (:type :array
                                    :items (:type :string))))
          :required ("answer")
          :additional-properties nil)
 :blocking-p t)
#+END_SRC

The reader and validator deliberately reject reader labels, circular or shared
structure, extra forms or trailing garbage, unknown or duplicate keys, improper
lists, and values of the wrong type. =:tools= is =nil=, =:all=, or a proper list
of canonical tool-spec strings. =nil= grants no ordinary tools; =:all= grants
all child-safe tools. List entries are exact =namespace.name= values,
=namespace.*= patterns, or =web_search=. Unknown or unavailable tools are
rejected. Role files cannot grant =self.*=, =task.*=, =job.*=, or =yield.*=
directly. Delegation and the terminal yield tool come only from the task
runtime's policy.

=:spawns= is =nil=, =:all=, or a proper list of role-name strings. =nil= denies
child delegation, and every other value remains subject to the depth limit.
=:models= is =nil= or a nonempty proper list of known model identifiers or
supported =@= aliases. =:reasoning-effort= is =nil=, =:auto=, or a supported
effort keyword. =:blocking-p= is exactly =t= or =nil= and defaults to =nil=.
There are no wildcard strings, scalar-list conveniences, or alternate key
spellings.

=:output= is =nil= or the native schema DSL shown above: a property list with
keyword type names and string-keyed object properties. It recursively supports
=:object=, =:array=, =:string=, =:number=, =:integer=, =:boolean=, =:null=, and
=:enum=. A successful child must submit matching structured data when a schema
is present. In schemas and structured values, =nil= represents JSON false and
=:null= represents JSON null where the distinction matters. JSON Schema is used
only when translating values at a provider or API compatibility boundary; role
files and internal task results remain native Lisp data.

=task.run= starts one child from flat =agent=, =task=, optional =name=,
=context=, and =async= arguments, or starts an independent =tasks= batch with
shared nonempty =context=. Synchronous children are collected into the tool
result. =async: true= detaches a non-blocking role and returns a job identifier;
a role with =:blocking-p t= always remains synchronous. Every child must end
with exactly one =yield.submit= carrying =success=, =failed=, or =aborted=. A
normal model stop without that yield is a failed child run.

A synchronous =task.run= keeps the primary model at that tool call until every
synchronous child finishes. Children in one batch may run concurrently through
the reusable pool; separate provider tool calls are still executed in order.
Detached children continue while the primary model proceeds, and the responsive
composer remains usable in either case. The default pool admits up to eight
running children, configurable with =AUTOLITH_TASK_MAX_CONCURRENCY= up to the
implementation maximum of 32.

Queued and running children appear in stable creation order in a colored live
strip immediately above the modeline. Running children share a spinner updated
four times per second, queued children have a static marker, and each row shows
the child identity, role, detached state, and current tool or assignment. The
strip shows at most eight children and summarizes any overflow. Terminal
children disappear without entering conversation history.

=job.list= lists visible jobs, =job.get= inspects one job and its progress,
=job.wait= waits for a bounded interval, and =job.cancel= interrupts a queued or
running job and its descendants. The primary agent sees only jobs owned by its
current conversation. A child sees only descendants of its own task identity,
never its parent, siblings, unrelated task trees, or another conversation's
jobs. Child transcripts and complete result artifacts remain in private task
storage outside the normal conversation picker.

=/compact on=, the default, hides successful routine tool results while retaining
failures, file-write and file-edit results, and arbitrary execution results from
=shell.run=, =lisp.eval=, and =self.eval=. =/compact off= shows every tool result.
The choice shares the private global preferences file with =/trace=. This changes
terminal presentation only; every result remains in durable conversation history
and provider context. With no argument, =/compact= still forces context compaction.

Context compaction keeps long conversations going. When the newest provider
step reports total token usage at or beyond the compaction limit, Autolith asks
the model for a context checkpoint summary, appends it as a durable
=(:summary ...)= record, and replaces the request projection with one bridge
message carrying that summary; the durable conversation file keeps everything.
The 5.6 family models have a 272000 token context window (read from the live
model catalog), and compaction triggers at 80 percent of the window by
default. Set =AUTOLITH_COMPACTION_THRESHOLD= to a percentage between 1 and 95 to
compact earlier or later, and =AUTOLITH_CONTEXT_WINDOW= to override the window
for unlisted models. =/compact= with no argument forces a compaction immediately, and
=/status= shows how full the context currently is.

=/status= reports the model, token usage, and subscription rate limit windows
parsed from provider responses. Requests never ask for the provider's priority
service tier, so Autolith always uses the standard path that drains rate limits
slowest. Hosted web search is requested in cached mode by default; set
=AUTOLITH_WEB_SEARCH= to =live= or =disabled= to change it.

=/later INPUT= durably schedules ordinary input for the next usable rate-limit
reset. An exhausted weekly window takes precedence over the five-hour window.
When the provider temporarily omits its primary window, Autolith estimates five
hours without scheduling past a reported secondary reset. The queue survives
restart in =${XDG_STATE_HOME:-~/.local/state}/autolith/later.sexp=, retains the
workspace in which each input was scheduled, and retries after handled provider
failures. =/later= lists the queue and =/later cancel ID= removes one entry.

Use a double slash to send a user message that begins with a literal slash.

* User Lisp configuration

Autolith loads the optional
=${XDG_CONFIG_HOME:-~/.config}/autolith/init.lisp= after its tracked system and
selected private image commit, before it creates the provider request path. The
file is read in the =AUTOLITH= package and is loaded again when reconnecting a
retained generation, so it remains the final user-controlled configuration
layer. It is ordinary Common Lisp running with the user's full privileges.

This entry point is distinct from =overlays/*.lisp=, which exist only to migrate
old self-modifications. User configuration is never built into the pristine
recovery image.

The request-local context registry is available from this file. A contributor
receives a read-only request snapshot and returns no contribution, one
contribution, or a list. For example:

#+BEGIN_SRC lisp
(define-context-contributor my-context-advice (request)
  "Return local advice when the newest user message requests a release."
  (let ((input (request-context-latest-user-text request)))
    (when (and input (search "release" input :test #'char-equal))
      (make-context-contribution
       :identifier "release-check"
       :instruction "Verify the release artifacts before publishing."
       :lifetime ':turn
       :priority 40))))
#+END_SRC

Contributions stack unless they declare deduplication, supersession, or a
conflict group. Priority controls ordering and admission to the bounded advice
budget rather than selecting one global winner. The rendered block exists only
in its provider request and never becomes conversation or memory content.
=/context= lists registrations, the last active and omitted contributions, and
bounded contributor failures.

=define-context-contributor= is a defining form understood by =self.redefine=,
=self.discard=, and private mutation replay. Autolith and =init.lisp= can
therefore use the same definition and registration mechanism. The lowercased
function name is its stable registry identifier. The lower-level
=register-context-contributor= function remains available when registration
must be managed separately.

Interactive commands use the same live extension model:

#+BEGIN_SRC lisp
(define-application-command my-version-command
    (:name "/my-version"
     :argument nil
     :description "show my local integration version"
     :tip "shows the version supplied by init.lisp."
     :busy-behavior :inspect
     :terminal-behavior :shared)
    (application invocation)
  (declare (ignore invocation))
  (application-present application "integration 3")
  :continue)
#+END_SRC

The stable Lisp definition name identifies revisions even when slash names or
aliases change. =:inspect= permits only an argument-free invocation while a
turn is active; =:hold= waits for the main application thread, and =:cancel=
requests shutdown. Terminal behavior is =:shared=, =:exclusive=, or
=:exclusive-without-arguments=. Every command requires a nonblank tip. The
defining form participates in =self.redefine=, =self.discard=, private replay,
dynamic completion, help, and startup tips.

Conversations are append-only readable forms under
=${XDG_DATA_HOME:-~/.local/share}/autolith/conversations/=. A truncated final form is
ignored after a crash; malformed complete records are rejected. If a crash
leaves a complete provider function call without its correlated tool result,
loading appends an explicit unknown-outcome failure and pairs it with that call
before replay. Autolith never assumes that the interrupted tool had no effects.

* Persistent memories

Autolith stores persistent memory in the readable append-only file
=${XDG_DATA_HOME:-~/.local/share}/autolith/memories.sexp=. It does not require
SQLite. Each =(:memory ...)= record is a complete value; replacing a memory
appends another complete record with the same stable identifier. A
=(:memory-forgotten ...)= record removes that identifier from active recall.
The reader binds =*read-eval*= to =nil=. It ignores an incomplete final form
after a crash and atomically repairs that tail before the next append.

Memories are workspace-scoped by default. They are recalled only when Autolith
runs in the same workspace. Global scope is intended for cross-workspace user
preferences and durable guidance. =memory.list= shows metadata, =memory.search=
ranks case-insensitive query terms across title, body, tags, and workspace,
favoring title, tag, and broad-coverage matches. =memory.read= returns one
complete active memory.
=memory.remember= creates a memory or replaces one by identifier.
=memory.remember= limits the complete body to 5,000 characters so one
=memory.read= call can return it without hidden paging.
=memory.forget= stops active recall by appending a tombstone. Because the log is
append-only, forgetting is not secure erasure of historical bytes; stop
Autolith and deliberately rewrite or remove the private file when erasure is
required.

Immediately before each normal provider request, Autolith ranks global and
current-workspace memories against the latest user input. When matches exist, a
bounded request-local contribution supplies quoted JSON metadata, tags, and
short excerpts. It never enters durable conversation history. The model uses
that notice to decide whether to call =memory.search= and =memory.read= for exact
recall. A malformed log does not prevent an ordinary model request: recall is
omitted and the failure appears in =/context= diagnostics, while memory tool
calls report the typed persistence failure.

Autolith does not automatically summarize every conversation. The model writes
a memory when durable information is likely to help later, and the prompt tells
it never to store credentials, secrets, transient progress, or guesses. This
keeps memory creation visible in conversation tool history and avoids extra
background model traffic.

* Workspace agendas

Autolith stores short task and note lists in the readable atomic file
=${XDG_DATA_HOME:-~/.local/share}/autolith/agendas.sexp=. Each agenda is keyed by
the canonical workspace directory and contains at most 32 stable-ID items with
=todo=, =doing=, =blocked=, =done=, or =note= status. The reader disables
=*read-eval*=, and each successful update publishes the complete versioned state
atomically with mode 0600.

The current workspace agenda is included in full as quoted untrusted data on
every provider request. =agenda.list= reads it, =agenda.add= creates an item,
=agenda.update= changes text, status, or attached memory IDs by item ID, and
=agenda.remove= deletes an item. Each item may attach at most 16 active
persistent memories by stable ID. Replacing a memory keeps every attachment
pointing at its newest value. Supplying an empty =memory-ids= array detaches all
memories from an item. Existing version-one agenda files load with no attachments
and upgrade atomically on their next mutation.

=agenda.transport= lists all known workspace keys, views another agenda, copies
one agenda into an existing target workspace, or moves it by merging into the
target and removing the old key. The source path may already be absent, so a
repository that moved before transport remains recoverable. Copy and move
preserve target items and memory attachments, combine attachments on exact
text-and-status duplicates, and give copied items fresh IDs only when an ID
collides.

* Workspace, search, memory, agenda, Lisp, and self tools

The =fs.*= and =shell.*= tools cover everyday workspace work: =fs.read=
returns numbered file lines from an optional window, =fs.list= lists one
directory with entry kinds and sizes, =fs.write= creates or replaces a file,
=fs.edit= replaces exact text with a uniqueness check, and =shell.run=
executes one external command with a bounded timeout and combined output.
Unknown commands ask before execution. Allowing once or permanently runs the
command with an isolated network and a read-only host filesystem except for the
workspace and temporary directories. A permanent approval matches the exact
command text and canonical working directory and is stored privately in
=${XDG_STATE_HOME:-~/.local/state}/autolith/permissions.sexp=. Use
=/permissions= to return to ask mode, allow sandboxed commands for the session,
let commands run with full user privileges for the session, list exact saved
approvals, or clear them. Corrupt permission state is ignored with a warning and
all unknown commands continue to ask.

These tools run in the active process but never modify the Lisp heap, so the
model does not need =self.eval= to look around. The stable launcher and recovery
artifacts stay read-only through the filesystem tools and ordinary command
sandbox. This boundary reduces accidental damage from external programs. It
does not make Autolith as a whole a security sandbox.

The =search.*= namespace is the normal way to discover workspace code. It uses
one warm [[https://github.com/dmtrKovalenko/fff][fff]] index shared by all search tools. =search.files= fuzzy-matches file
and path names, =search.glob= filters paths with one glob, and =search.content=
supports plain, regex, or fuzzy content matching with result, per-file, context,
time, and file-size bounds. =search.multi-content= performs one SIMD-backed OR
search for several literal alternatives. Content queries accept inline fff
constraints such as =*.lisp symbol=, =src/ symbol=, and =!tests/ symbol=.
Responses use workspace-relative locations, compact metadata, and explicit
pagination cursors so the model can read a likely file instead of repeating
slightly different shell searches.

Every executed tool result records and displays CPU and real elapsed time. Real
time includes authorization, external process duration, and other waits around
the complete tool operation; CPU time measures work consumed by the Autolith
process. Calls rejected before execution and results synthesized during crash
repair have no invented timing.

Bootstrap builds fff revision
=44a5b259570730a4236ecbf06673d43ef7b2263e= from its locked Rust source and
publishes =libfff_c.so= under
=${XDG_DATA_HOME:-~/.local/share}/autolith/native/fff/=. The library is loaded
through its C ABI inside Autolith and is never installed on =PATH= or into the
system library directories. Its watched content index and LMDB search metadata
live below =${XDG_CACHE_HOME:-~/.cache}/autolith/fff/=. Autolith stops the native
watcher before checkpoints and clean exit, then recreates it lazily when search
is next used. If the isolated helper fails, Autolith discards only the
rebuildable ranking and history databases before retrying it once.
=AUTOLITH_FFF_LIBRARY= may point at a compatible development build; ordinary
installations should leave it unset.

=AGENTS.md= files are discovered the way Codex discovers them: Autolith walks up
from the workspace to the nearest =.git= project root, then includes every
=AGENTS.md= from the root down to the workspace, root first, bounded and
refreshed on every request. Without a project root only the workspace file
is considered.

An optional =AUTOLITH.org= lives only at that project root. It is a voluntary
adaptation ledger describing recurring project friction, active Autolith
adaptations, their scope and verification, and when they should be removed.
Its bounded contents supplement =AGENTS.md= as request-local advisory context;
they are never executed as Lisp and never override repository policy. Presence
also activates a short observable context contribution encouraging useful,
project-scoped self-improvement when concrete evidence warrants it. The ledger
and reminder never enter durable conversation history, memory, or compaction.

The =lisp.*= tools manage named persistent SBCL REPLs through a small readable
S-expression protocol. =eval=, =compile=, =load-system=, =describe=, =source=,
and =run-tests= accept a =repl= name and default to =default=. A missing REPL
starts from pristine SBCL. =start= creates one from pristine or a compatible
saved image, =reset= deliberately replaces one from a selected image, =stop=
forgets one, and =repls= shows the live pool. Separate REPLs never share heap
objects with each other or the active agent, so pristine and instrumented
behavior can be compared side by side.

=lisp.source= uses SBCL's live definition metadata to map functions, macros,
compiler optimizers, source transforms, transforms, VOPs, and IR1 converters
back into the hash-verified matching source tree. It returns the complete
recorded top-level form when debug offsets permit and otherwise returns a
numbered source window. The installed source tree remains unchanged.

=lisp.save-image= forks a saver from a selected REPL without stopping its
parent, boot-probes the unpublished core with the pinned executable, then
atomically publishes the core, manifest, and required note. =lisp.images= lists
the immutable image names, parentage, compatibility, and notes. The model sees a
bounded reminder of these notes on every request and can choose a saved image
only when starting or resetting a REPL. The virtual =pristine= image cannot be
overwritten.

The =self.*= tools inspect bindings, read tracked definitions or matching SBCL
source, summarize active mutation and recovery state, evaluate, redefine, set,
exercise or discard pending changes, persist complete Autolith definitions,
show effective live changes, privately commit them, checkpoint, list
generations, and request immediate rollback. =self.redefine= accepts an explicit
active package, temporarily unlocks a locked implementation package while
reading and installing the form, restores the lock, and journals the canonical
package. =self.commit= preserves that package in its replay entry, allowing an
SBCL Lisp-level definition to be reconstructed without changing the runtime
executable.

Durable self-modifications never patch the tracked repository. =self.redefine=
and =self.set= journal successful reconstructible changes in the active image;
=self.diff= coalesces repeated edits and shows the effective changes still
pending in that image lineage. =self.exercise= journals a focused pass or
failure against one mutation, while =self.discard= restores its exact preceding
live state. =self.commit= checks all pending changes and writes an immutable full
snapshot under
=${XDG_DATA_HOME:-~/.local/share}/autolith/image-commits/COMMIT-ID/=. Each snapshot
contains =manifest.sexp= and executable =reconstruct.lisp=. Before selection,
Autolith loads the script in a clean pinned source process. It then copies the
artifacts into a separate Git repository under
=${XDG_STATE_HOME:-~/.local/state}/autolith/mutation-history/= and commits them
using the private commit title. The atomic selection record is
=${XDG_STATE_HOME:-~/.local/state}/autolith/current-image-commit.sexp=. Normal
startup loads that script after the tracked system. If the selected canonical
manifest or replay script was deleted, startup restores it from the exact Git
object before loading it. =self.persist-definition= performs the same complete
transaction directly for one definition. Legacy =overlays/*.lisp= files load
when no private commit exists and are imported by the first private commit.

Running Autolith with its own repository as the workspace turns it into an
ordinary development agent for Autolith: =fs.write= and =fs.edit= then work on
the tracked source like on any project, with only the stable launcher and
recovery artifacts read-only. From any other workspace those tools refuse
Autolith's own source entirely. =self.commit= has no workspace Git behavior and
works identically from any workspace because every file it writes belongs to
Autolith's private data or state root. Its Git commits belong only to the
private mutation-history repository. Workspace repository commits are handled
through ordinary workspace commands only when the user requests them.

Autolith keeps all project definitions in one =AUTOLITH= package. Files and CLOS
protocols express subsystem boundaries without scoped packages.

* Generations and recovery

=/checkpoint= first requires a clean Git worktree and a passing =./script/check=. It
briefly excludes live mutation, revalidates the source revision, and forks a
coordinator. The active parent resumes immediately. A saver child writes a
temporary core. Before publication, the coordinator boots that core through a
private probe and requires its exact generation ID, tracked source commit, and
private image and mutation-history Git commits. Only then does it publish the
core, manifest, and selected-generation record. Before the fork, checkpoint
preparation checks and commits any pending reconstructible mutations.
Every new generation directory receives a complete =reconstruct.lisp=, and its
version-3 manifest records that path and both private commit identities captured
by the saved core. Manifest versions 1 and 2 remain readable.

=/rollback ID= atomically selects a compatible generation and leaves the active
process through a dedicated recovery status. The stable launcher then starts the
matching core and source revision immediately.

List or boot retained generations with:

#+BEGIN_SRC sh
autolith --recovery --list
autolith --recovery --generation GENERATION-ID
#+END_SRC

Recovery normally runs from the separately built pristine core without loading
the active source or damaged active core. Its manifest records exact build-input
provenance, and both installation and startup require a versioned core handshake.
An absent or invalid pristine image falls back to the source launcher. Recovery
sanitizes all terminal output, inspects pending
mutation identities, creates a private detached Git checkout at the manifest's
recorded commit, and points the resumed application at that source. A fatal
active-path failure writes a private allowlisted crash capsule, publishes it to
that launcher instance, exits with status 70, restores the terminal, and boots
the selected generation. If that generation fails fatally, recovery tries other
compatible retained generations newest first, restoring terminal state and
refreshing crash context between attempts. If no compatible retained generation
exists or all of them fail, recovery creates a private clean checkout of the
current committed source and boots it with the locked dependency setup.
Before inspection begins, the pristine core prints a red =RECOVERY IMAGE= label
and explains that it exists to avoid loading the damaged active core.

Worker images are never candidates for active startup, rollback, or recovery.
Saving or loading an instrumented worker core does not replace the pinned SBCL
executable, the preloaded active baseline, or the pristine recovery core.

Automatic recovery forwards the original application arguments while marking
the resumed process as recovered, restores the capsule's conversation, and
retains the last rendered sequence so ordinary terminal scrollback is not
duplicated. An explicit =resume ID= command overrides checkpoint-time
conversation state.

Saved cores are compatible only with the recorded SBCL version, operating system
build, and architecture. Source remains authoritative for rebuilding the base.
To reconstruct a generation without its core, check out the manifest's
=:git-commit=, materialize the locked dependencies with =./script/bootstrap=,
load the =autolith= system, then load the generation's =reconstruct.lisp= in the
=AUTOLITH= package. The script is complete and ordered; the manifest and
=autolith --recovery --list= show its exact pathname.

After loading the locked project setup from that checkout, the essential Lisp
sequence is:

#+BEGIN_SRC lisp
(asdf:load-asd (truename "autolith.asd"))
(asdf:load-system :autolith)
(load #P"/path/from/generation/manifest/reconstruct.lisp")
#+END_SRC

* Security status

Autolith is not a security sandbox. It executes model-generated Lisp and external
commands with the privileges of the user running it. Process boundaries,
restricted tool surfaces, append-only records, Git, and retained generations
reduce accidental permanent damage; they do not contain malicious code.

Credentials are excluded from source, conversations, memories, journals,
manifests, crash capsules, tool output, and saved cores. Provider objects retain
credential paths, not token values, and checkpointing is rejected inside a
credential request scope.
