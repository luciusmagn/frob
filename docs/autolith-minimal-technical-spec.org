#+TITLE: Autolith - Minimal Technical Specification
#+SUBTITLE: A live, self-modifying Common Lisp agent
#+STARTUP: content
#+OPTIONS: toc:2 num:nil

* Status

Initial implementation target:

- Linux x86-64
- SBCL
- Terminal interface
- Single primary agent
- No hostile-code sandboxing

macOS is the next platform target, followed by Windows, BSD systems, and
additional Linux architectures.

* 1. Purpose

Autolith is a small Common Lisp agent that can inspect and modify its own
running image.

Ordinary changes take effect immediately without restarting the agent.
Autolith saves working generations so that it can resume or roll back after
a damaging change. A separate pristine recovery image remains available
when the active agent can no longer repair itself.

The source repository must remain sufficient to rebuild an image from
scratch.

* 2. Design Principles

- Keep the initial codebase small.
- Prefer Common Lisp over generated Python or shell scripts.
- Treat the live Lisp image as the primary runtime environment.
- Make self-modification explicit through a small set of tools.
- Preserve source, conversations, and working generations.
- Use process isolation for reliability, not as a security claim.
- Keep platform-specific behavior behind narrow adapters.

* 3. Runtime Layout

Autolith consists of four principal parts.

** 3.1 Launcher

The executable command is =autolith=.

The launcher is small, stable, and outside the mutable agent image. It:

- Starts the current active image.
- Validates a source-built preloaded image before normal startup.
- On an interactive launch, offers to bootstrap a missing or stale preloaded
  image before starting and falls back to source when the user declines.
- Falls back directly to source loading for a non-interactive launch.
- Recognizes =--from-source= and bypasses image validation, bootstrap prompts,
  and missing-image notices when it is present.
- Recognizes =--recovery= itself.
- Starts the pristine recovery image when =--recovery= is present.
- Forwards all remaining command-line arguments.
- Detects fatal exits from the active image.
- Passes available crash context to the recovery image.
- Restores terminal state after an abnormal exit.

The launcher and pristine recovery artifacts are not writable through
normal =self.*= tools.

** 3.2 Active Agent

The active SBCL image contains:

- The model client and agent loop.
- Conversation and working state.
- Persistent memory selection and storage.
- Workspace agenda selection and storage.
- Tool definitions and dispatch.
- The terminal interface.
- Self-inspection and self-modification facilities.
- Loaded Lisp dependencies.

After the tracked system and selected private image commit load, Autolith loads
=${XDG_CONFIG_HOME:-~/.config}/autolith/init.lisp= when it exists. This
user-authored Common Lisp is the final executable configuration layer and
therefore takes precedence over built-in and privately committed definitions.
It executes with the user's full privileges, is loaded again when reconnecting
a retained generation, and is never included in the pristine recovery image.

This process is expected to evolve while it is running.

Setup may install a preloaded baseline active image containing compiled Autolith
code and dependencies, but no application state or credentials. Its manifest
identifies every compiled Autolith source input, the dependency lock, and the
exact runtime. The launcher boots it only after its private probe verifies those
inputs against the current source tree. It is distinct from retained working
generations and from the pristine recovery image. Source loading remains the
authoritative fallback.

** 3.3 Lisp Workers and Images

The =lisp.*= tools operate in named, persistent SBCL REPL processes managed as
a pool. Each REPL starts from either the pristine pinned SBCL image or one
compatible saved worker image selected explicitly by the agent. REPLs do not
share heap state with each other or the active agent and cannot modify another
image through ordinary object references.

A saved worker image contains the complete Lisp heap of one REPL, including
intentional changes to SBCL's Lisp implementation. It never replaces or
modifies the pinned low-level SBCL runtime executable. Its immutable manifest
records its identity, parent image, exact runtime, creation time, and a durable
note explaining the changes and intended use. The pristine image is a virtual,
immutable root and cannot be overwritten.

The active agent receives a bounded inventory of saved worker-image notes on
every model request. It can list compatible images and live REPLs, compare a
modified REPL with a pristine one, and choose an image whenever it starts or
resets a named REPL. A worker image is never loaded by the pristine recovery
process.

Workers may use the bundled Lisp systems, Quicklisp distribution, matching SBCL
source, and installed external programs. They are used for:

- Fresh REPL evaluation.
- Compiling experimental code.
- Loading packages.
- Running tests.
- Inspecting behavior outside the active image.

** 3.4 Recovery Image

The pristine recovery image is a stable SBCL image containing only the
facilities needed to diagnose, repair, rebuild, select, and boot agent
generations.

It is not the normal execution environment.

* 4. Model Access

Autolith implements its own model client and agent loop.

The initial provider supports Codex subscription access through the
user's ChatGPT authentication without bundling or launching the Codex
CLI.

Autolith owns:

- System instructions.
- Conversation state.
- Context selection and compaction.
- Tool schemas and dispatch.
- Streaming presentation.
- Retry and fallback policy.
- Token and usage accounting available from the provider.

Provider-specific authentication and transport behavior are isolated
behind a replaceable interface.

Local image input follows the provider's Codex-compatible content shape. The
terminal composer represents attachments with numbered =[Image #N]= labels and
submits each image before the accompanying text. Supported local PNG, JPEG,
GIF, and WebP data is validated and reduced to the provider's high-detail image
limits before use. Portable conversation records contain image descriptors,
not inline binary or base64 data. Immutable private artifacts remain under the
data root and are reconstructed into request-local data URLs during replay.

The read-only =fs.view-image= tool provides the other half of local image
support. It accepts an absolute or workspace-relative image pathname, validates
and privately preserves the image through the same preparation pipeline, and
returns native =input_image= content inside its correlated function-call output.
This lets the model inspect images created or discovered during a turn without
using OCR or lossy text approximations. Its durable conversation record retains
only a portable artifact descriptor and reconstructs the image output on replay.

Credentials are stored outside:

- Saved Lisp cores.
- Conversation files.
- Memory files.
- The Git repository.

An API-key provider may exist as an optional fallback, but is not the
primary access path.

** 4.1 Request-Local Context

Autolith assembles ephemeral context contributions immediately before each
normal provider request. A contribution has a stable identifier, trusted
instruction, optional untrusted evidence, source, lifetime, priority, and
optional deduplication, supersession, or conflict metadata. Contributors
receive a read-only request snapshot and may return any number of compatible
contributions.

Compatible contributions stack. Priority controls rendering order and decides
which advice survives the bounded request budget; it is not a winner-takes-all
selection rule. Equivalent contributions collapse by explicit deduplication
key, specific advice may explicitly supersede generic advice, and only an
explicit conflict group selects one strongest value. Mandatory contributions
do not compete for the advisory budget.

Rendered contributions are provider-request input only. They are never appended
to conversation files, incorporated into compaction summaries, offered to
memory creation, or restored as conversation history. A next-request
contribution is consumed only after a completed provider response. State-bound
contributors recompute their output for every request. Non-model diagnostics
retain contributor identifiers, selection, omission, and bounded failure
reports so the mechanism remains observable.

The same registration API is available to tracked code, private
self-modifications, and the user initialization file. A failing contributor is
reported and omitted without preventing unrelated provider work.
=define-context-contributor= is a complete supported self-modification defining
form: exploratory installation registers it, discard restores both its function
binding and prior registration, and private commit replay reconstructs both.

* 5. Tool Model

Tools are registered internally. MCP-compatible exposure may be added,
but in-process tools do not require loopback MCP transport.

** 5.1 =lisp.*=

=lisp.*= tools operate only in named, heap-isolated Lisp workers. Operations
that evaluate, compile, load, describe, inspect source, or run tests accept a
REPL name and default to =default=. A nonexistent REPL starts from pristine
SBCL. An explicit start or reset may instead select any compatible saved worker
image. An existing REPL never changes its base image implicitly.

Initial operations should include:

- =lisp.eval=
- =lisp.compile=
- =lisp.load-system=
- =lisp.describe=
- =lisp.source=
- =lisp.run-tests=
- =lisp.start=
- =lisp.stop=
- =lisp.reset=
- =lisp.repls=
- =lisp.images=
- =lisp.save-image=

=lisp.save-image= forks a saver from the selected REPL, validates the saved
core with the pinned runtime, and atomically publishes its core, manifest, and
note without stopping the parent REPL. Saved images may include Lisp-level SBCL
changes and ordinary experimental heap state.

** 5.2 =self.*=

=self.*= tools operate on the active Autolith image.

Initial operations should include:

- =self.inspect=
- =self.source=
- =self.eval=
- =self.redefine=
- =self.set=
- =self.persist-definition=
- =self.status=
- =self.discard=
- =self.exercise=
- =self.diff=
- =self.commit=
- =self.checkpoint=
- =self.generations=
- =self.rollback=

The exact tool set may remain small. General operations should be
composed from ordinary Lisp wherever practical.

Starting with =--immutable= omits =self.eval=, =self.redefine=, =self.set=,
=self.persist-definition=, =self.discard=, =self.exercise=, =self.commit=,
=self.checkpoint=, and =self.rollback= from both dispatch and the provider
schema. Inspection through =self.inspect=, =self.source=, =self.status=,
=self.diff=, and =self.generations= remains available. The system prompt must
describe the reduced capability accurately. Recovery preserves the flag, while
a later launch without it restores the normal tool registry.

** 5.3 =memory.*=

=memory.*= tools operate on durable user-specific memories stored outside
conversations and saved cores. Initial operations include:

- =memory.remember=
- =memory.list=
- =memory.search=
- =memory.read=
- =memory.forget=

Memories are workspace-scoped by default. Global memories are reserved for
cross-workspace preferences and guidance. Immediately before a normal provider
request, Autolith ranks global and current-workspace memories against the latest
user input. When related values exist, the model receives a bounded
request-local notice with identifiers, metadata, and short excerpts, then uses
the search and read tools for exact recall. The notice is an ephemeral context
contribution and its values are untrusted quoted data, not instructions. Memory
writes must exclude credentials, secrets, transient progress, and unsupported
guesses.

** 5.4 =agenda.*=

=agenda.*= tools operate on short durable agendas keyed by canonical workspace
directory. Initial operations include:

- =agenda.list=
- =agenda.add=
- =agenda.update=
- =agenda.remove=
- =agenda.transport=

Each item has a stable identifier, bounded complete text, lifecycle or note
status, creation and update times, and an optional bounded list of stable memory
identifiers. Replacing a memory preserves every agenda attachment because its
identifier remains stable. Every model request receives the current workspace
agenda in full as quoted untrusted data. Transport enumerates and inspects other
workspace agendas, copies them into existing target directories, and moves them
to new directory keys even when the old source directory no longer exists. Copy
and move merge without discarding existing target items or memory attachments.

** 5.5 Workspace Commands

=shell.run= requests user authorization before every unrecognized command. An
approval is defined by exact command text and canonical working directory. The
user may save that exact approval or choose a session-wide mode through
=/permissions=.

Ordinary approvals run the command in a Linux process sandbox with an isolated
network, a read-only host filesystem, writable workspace and temporary roots,
protected repository metadata, process namespaces, =no_new_privs=, and seccomp.
The user may explicitly grant full user privileges for the current session.
Malformed persistent permission data fails closed.

This command boundary limits accidental damage from external programs. It does
not make Autolith a hostile-code sandbox because filesystem, Lisp worker, and
active-image tools remain separately privileged capabilities.

** 5.6 Child Agents and Jobs

The primary agent may delegate bounded work to in-process child sessions. A
child uses the ordinary provider, streaming, tool-execution, authorization,
compaction, and turn-completion path. It does not use a second reduced agent
loop. Each child receives a separate conversation, provider configuration,
tool registry, explicit assignment, optional shared context, role instructions,
and depth.

Child conversations and result artifacts are private execution records beneath
the data root. They are not ordinary resumable conversations and never appear
in conversation selection. Each execution receives a unique artifact directory.
Terminal publication writes at most one portable result artifact and retains
its pathname in the parent-facing result.

Role definitions are bounded =*.sexp= files containing exactly one top-level
Common Lisp property list. Reading binds =*read-eval*= to =nil=, uses a fresh
=(copy-readtable nil)=, and requires clean end-of-file after that form.
Discovery precedence is project
=.autolith/agents/=, then
=${XDG_CONFIG_HOME:-~/.config}/autolith/agents/=, then bundled roles. A higher
precedence definition replaces the same lower precedence name. Parsing and
policy validation fail closed: malformed reader input, reader labels, circular
or shared structure, extra forms or trailing garbage, unknown or duplicate
keys, improper lists, invalid types, and invalid policy values are reported
rather than guessed. A malformed higher precedence role must not silently
expose or select a lower precedence role of the same name.

The exact role keys are =:name=, =:description=, =:instructions=, =:tools=,
=:spawns=, =:models=, =:reasoning-effort=, =:output=, and =:blocking-p=.
=:name=, =:description=, and =:instructions= are required nonempty strings.
=:tools= is =nil=, =:all=, or a proper list of canonical tool-spec strings.
=nil= grants no ordinary tools, while =:all= grants every child-safe tool, not
every primary-agent tool. Every explicit grant is intersected with the
child-safe registry. Canonical specs are exact =namespace.name= values,
=namespace.*= patterns, or =web_search=. Unknown or unavailable tools and every
=self.*=, =task.*=, =job.*=, or =yield.*= grant are rejected. External commands
retain the primary session's authorization boundary.

=:spawns= is =nil=, =:all=, or a proper list of role-name strings. =nil= denies
further delegation. An explicit role list or =:all= remains subject to the
configured maximum child depth. =:models= is =nil= or a nonempty proper list of
supported model identifiers or aliases. =:reasoning-effort= is =nil=, =:auto=,
or a supported effort keyword. =:blocking-p= is exactly =t= or =nil=.
The primary session owns its conversation's root jobs. A child owns only jobs
created below its own task identity, so its =job.*= tools cannot inspect or
control its parent, siblings, unrelated trees, or jobs from another primary
conversation.

=task.agents= is the read-only discovery interface. It reports only roles the
caller may select, including effective name, description, source, pathname,
model and effort policy, tool and spawn policy, and whether an output contract
exists. It also reports diagnostics needed to correct rejected role files. The
model should inspect it before choosing a specialized role when the available
roles or their restrictions matter.

=task.run= accepts one assignment or an independent batch. Synchronous children
hold the call until their terminal results are collected. A non-blocking role
may instead run as a detached job and return its identity immediately. A role
with =:blocking-p t= always remains synchronous. =job.list=, =job.get=,
=job.wait=, and =job.cancel= expose only jobs owned by the requesting session as
described above. Cancellation cascades from a job to its descendants.

Admission is bounded by batch and live-job limits. A bounded FIFO feeds a
reusable worker pool with configured concurrency and an implementation maximum.
Nested synchronous delegation must make progress even when every reusable
worker is occupied; it must not deadlock waiting for another worker. Runtime
deadlines use one shared monitor rather than one watchdog thread per job.
Terminal jobs have bounded retention, and terminal publication is single-writer
even when completion, cancellation, and timeout race.

Task workers and monitors are application runtime resources. Clean shutdown and
conversation replacement cancel outstanding work and join the pool. Checkpoint
preparation closes the pool before enforcing the single-threaded saver boundary.
The forked saver detaches inherited task state, and a retained-generation
reconnect constructs fresh runtime resources rather than reviving captured
workers.

Every child must finish by calling =yield.submit= exactly once with =success=,
=failed=, or =aborted= status. A normal assistant stop without a yield is a
failed child run. Successful yields carry useful parent-facing text and, when
the role declares =:output=, structured data satisfying its native schema DSL.
Schemas are property lists whose type names are keywords and whose object
properties are string-keyed association lists. The recursive structural subset
supports =:object=, =:array=, =:string=, =:number=, =:integer=, =:boolean=,
=:null=, and =:enum=. In schema and structured-output values, =nil= represents
JSON false and =:null= represents JSON null where the distinction matters. JSON
Schema conversion happens only at provider and API compatibility boundaries.
Failed or aborted yields carry an explicit error.
Invalid or repeated yields are rejected. A synchronous batch succeeds only when
every synchronous child succeeds, while its result still identifies every
child, terminal status, error, structured data, and artifact.

Request-local context is scoped to the conversation making the provider
request. Concurrent primary and child requests never share a mutable current
request slot. Contributions are assembled at each provider-request boundary,
attributed to that conversation, and remain ephemeral under section 4.1.

Tool restriction, process separation, depth, and scheduling are
accidental-damage and reliability boundaries, not a hostile-code sandbox.

* 6. Live Self-Modification

Autolith may redefine functions, methods, classes, macros, and mutable
global state while running.

A restart is not required for ordinary modification.

There are two levels of change.

** 6.1 Exploratory Change

An exploratory =self.redefine= or =self.set= change modifies the active
image and appends a stable mutation identity to the journal.

It may be tested, reverted, refined, or discarded without immediately changing
durable reconstruction state. =self.status= summarizes running, selected,
pending, and retained state. =self.exercise= records a focused pass or failure
against one effective mutation. =self.discard= restores the exact preceding live
state and removes one effective mutation from the pending set. =self.diff=
coalesces repeated revisions of a target and shows only effective changes from
the selected private image. The append-only journal retains the raw history.

** 6.2 Durable Change

A durable change normally follows this sequence:

1. Record the intended mutation in the mutation journal.
2. Compile and install the change in the active image.
3. Run relevant checks.
4. Persist the complete effective mutation state as an immutable private
   image commit under the data root.
5. Load the complete replay script in a clean pinned Autolith process.
6. Commit the snapshot to Autolith's private mutation-history Git repository.
7. Atomically select the private image and its exact Git commit.
8. Mark the journal entry as durable.

Each private image commit has a manifest and a complete executable Lisp
replay script. An atomic pointer under the state root selects the commit
loaded after the tracked system at normal startup and records the exact private
Git commit retaining its artifacts. =self.commit= checks all pending
=self.redefine= and =self.set= mutations in the current image lineage and
persists their coalesced effective state. It rejects a pending stack that returns
exactly to its baseline. =self.persist-definition= performs the complete
transaction for one definition. Both tools create commits only in Autolith's
private mutation-history repository; neither tool writes or commits a workspace
repository.

Persisted definitions are complete Lisp forms rather than blind textual
patches.

The tracked source remains authoritative for clean rebuilds of the base
system. Private replay scripts reconstruct committed user-specific
extensions. Saved cores preserve the exact live state of a working
generation. Legacy overlay files are imported into the first private
image commit so existing durable changes remain reconstructible.

* 7. Conditions and Restarts

Autolith uses the Common Lisp condition and restart system for recoverable
failures.

Operations should establish useful restarts where practical, such as:

- Retry.
- Supply a value.
- Skip the operation.
- Recompile.
- Restore the previous definition.
- Abort the current tool call.

** 7.1 LLM-Assisted Handling

When the general model path is healthy, Autolith may present a condition to
the LLM together with:

- The condition type and report.
- The failed operation.
- A bounded backtrace.
- Relevant recent events.
- Available restarts.

The LLM may inspect the system, make a repair, or select a restart.

** 7.2 Deterministic Provider Handling

Provider failures that the LLM cannot diagnose because the provider is
unavailable are handled without invoking recovery when possible.

Examples include:

- Network timeouts.
- Authentication expiry.
- Rate limits.
- Temporary service errors.
- Unavailable models.

Possible responses include refresh, retry, backoff, fallback, or a
visible terminal error.

** 7.3 Fatal Control-Path Failure

Recovery is required when the general agent path is no longer
trustworthy, including failures in:

- The model client.
- Response parsing.
- The main agent loop.
- Tool dispatch.
- The condition broker.
- Critical conversation state.

LLM-assisted condition handling must have a recursion guard. A serious
condition raised while another condition is already being handled by
the LLM is fatal.

Before exiting, Autolith writes a best-effort crash capsule and terminates
with a distinguished recovery status.

* 8. Persistence

Autolith persists nine kinds of state:

- Source in Git.
- Conversations as S-expression files.
- Memories as an append-only S-expression log.
- Mutations in an append-only journal.
- Private image commits as manifests and complete Lisp replay scripts.
- A private Git history retaining complete image-commit artifacts.
- Working generations as saved Lisp cores and reconstruction scripts.
- Worker images as immutable SBCL cores with manifests and durable notes.
- User-authored executable configuration under the XDG config root.

** 8.1 Conversation Files

Each conversation is stored in a separate =.sexp= file.

A fresh conversation remains in memory until it has a durable record. The
header and first record are published together atomically, so starting and
leaving an empty conversation does not create a conversation file.

Interactive conversation selection groups sessions from the current working
directory above sessions from other directories. Each group is ordered from
newest to oldest file activity, and group headings are not selectable.

A conversation file is an append-only sequence of top-level
S-expressions. It is not wrapped in one enclosing list.

When a crash leaves a complete provider function-call record without a
correlated complete tool-result record, loading must append an explicit
unknown-outcome failure before replaying that history. The repaired provider
projection places the failure with its call even when later messages were
already appended. Recovery must not assume the interrupted tool had no effects
and must not retry it automatically.

Example:

#+BEGIN_SRC lisp
(:conversation
 :version 1
 :id "2026-07-11-001"
 :created-at 3992846378)

(:message
 :seq 1
 :time 3992846378
 :role :user
 :content "Inspect the current provider implementation.")

(:tool-call
 :seq 2
 :time 3992846380
 :id 1
 :tool self.inspect
 :arguments (:symbol run-provider))

(:tool-result
 :seq 3
 :time 3992846381
 :call-id 1
 :status :ok
 :value (:lambda-list (request)
         :source "src/provider.lisp"))

(:summary
 :seq 40
 :time 3992847000
 :through-seq 32
 :content "The provider transport was inspected and simplified.")
#+END_SRC

Records may represent:

- Conversation metadata.
- User, assistant, and system messages.
- Tool calls and results.
- Condition and restart events.
- Context summaries.
- Provider metadata and usage.
- References to external artifacts.

Conversation records contain only portable readable data. Arbitrary
functions, streams, packages, CLOS instances, and implementation
objects are not serialized directly.

Reading is performed with =*read-eval*= bound to =nil=.

An incomplete final form caused by a crash is ignored or truncated.
All complete preceding forms remain valid.

Large or binary artifacts are stored separately and referenced from the
conversation file.

Every executed tool result records nonnegative CPU and real durations. Terminal
presentation shows both durations. Calls rejected before execution and repaired
unknown-outcome records omit timing rather than inventing a duration.

Compact terminal presentation is enabled by default and may be disabled as a
durable global preference. It hides only successful routine tool-result entries.
Failures, file writes and edits, and arbitrary execution through =shell.run=,
=lisp.eval=, or =self.eval= remain visible. Presentation filtering never removes
records from conversation storage, request projection, or compaction input.

** 8.2 User Configuration

The optional =${XDG_CONFIG_HOME:-~/.config}/autolith/init.lisp= file is the
explicit entry point for user Lisp. Autolith loads it in the =AUTOLITH= package
after the tracked system and selected private image commit, before the first
provider request. A retained generation reconnect loads the current file again
instead of relying on a copy captured in its core.

The file is trusted executable code with the same user privileges as Autolith.
A load failure is a structured startup error identifying the file and original
condition. The file is separate from legacy mutation overlays and private image
commits, so mutation rollback never discards user policy.

** 8.3 Mutation Journal

The mutation journal is append-only and records enough information to
understand recent changes after a crash.

A journal entry should identify:

- Time.
- Target definition or state.
- Previous source or recoverable reference.
- Proposed change.
- Result.
- Related private image commit, if durable.
- Related mutation-history Git commit, when recorded.
- Related checkpoint, if saved.

The journal is for recovery and diagnosis. It does not replace private
image commits or their complete replay scripts.

** 8.4 Persistent Memories

Persistent memories are stored in one readable =memories.sexp= file beneath the
data root. The file begins with a versioned header and continues as an
append-only sequence of complete top-level forms. A memory record contains at
least:

- A stable identifier.
- Creation and replacement times.
- Global or workspace scope.
- The owning workspace for workspace scope.
- A short retrieval title.
- Complete bounded content.
- Retrieval tags.
- The source conversation identifier, when available.

Replacing a memory appends a complete record with the same identifier. A
tombstone removes an identifier from active recall without claiming secure
erasure of earlier bytes. Reading binds =*read-eval*= to =nil=, rejects malformed
complete records, and tolerates an incomplete final form. The next successful
append atomically repairs an incomplete tail before publishing the new record.

Memory search uses a deterministic weighted lexical rank. Title and tag matches
weigh more than body and workspace matches, broad term coverage receives an
additional weight, and update time plus identifier provide stable tie breakers.
Immediately before a normal model request, the same rank selects a bounded set
of global and current-workspace memories related to the latest user input. Their
metadata and short excerpts enter only the request-local context contribution;
exact content remains available through =memory.search= and =memory.read=.
Memory corruption omits the contribution, records a bounded contributor failure
for diagnostics, and does not prevent unrelated model work.

** 8.5 Workspace Agendas

Workspace agendas are stored in one readable =agendas.sexp= file beneath the
data root. The complete versioned state is published atomically with private
permissions. Reading binds =*read-eval*= to =nil= and rejects malformed forms,
duplicate workspace keys, duplicate item identifiers, unsupported statuses,
oversized item text, and overfull agendas.

One workspace agenda contains at most 32 ordered items. Each record carries the
canonical directory key and each item carries a stable identifier, complete
text of at most 500 characters, status, creation time, update time, and at most
16 unique persistent-memory identifiers. The current version-two writer reads
version-one items as having no attachments and upgrades the complete file on
the next successful mutation. Removing the final item removes the empty
workspace record. A malformed file degrades to an empty agenda with a structured
warning instead of preventing unrelated model work.

Transport accepts an absolute missing source key so an already-moved repository
can be recovered. Its target must be an existing directory. Copy retains the
source key; move removes it. Both merge with existing target items, combine
memory attachments on exact text-and-status duplicates, resolve identifier
collisions without replacing target items, and reject a result beyond either
the item or per-item attachment limit.

* 9. Generations and Checkpoints

A generation is a known working state consisting of:

- A saved Lisp core.
- The tracked base Git commit.
- A complete Lisp reconstruction script.
- The applicable private image commit, when one exists.
- The applicable mutation-history Git commit, when one exists.
- A small manifest.
- The applicable mutation-journal position.

The manifest identifies at least:

- Generation identifier.
- Core pathname.
- Reconstruction script pathname.
- Private image commit identifier, when applicable.
- Mutation-history Git commit, when applicable.
- Tracked base Git commit.
- SBCL build.
- Operating system.
- Architecture.
- Creation time.

Core files are not assumed to be portable across SBCL builds,
operating systems, or architectures.

** 9.1 Linux Checkpointing

On Linux, Autolith creates a checkpoint without stopping the active agent:

1. Briefly block live mutation.
2. Check and privately commit pending reconstructible mutations.
3. Write a complete reconstruction script in the new generation directory.
4. Flush source, conversation, and journal output.
5. Fork a child process.
6. Resume the parent.
7. Let the child detach inherited external resources.
8. Let the child save the Lisp core and exit.
9. Atomically publish the completed core and manifest.

The parent remains the active agent.

Checkpoint files are written under temporary names until complete.

Checkpoint logic must account for SBCL thread and external-resource
requirements before invoking =save-lisp-and-die=.

** 9.2 Other Platforms

Platforms without a validated fork-and-dump path may initially create a
checkpoint through a controlled restart.

This limitation must remain behind a checkpoint backend and must not
affect the Linux design.

** 9.3 Rollback

Rollback selects a retained working generation and starts its matching
core with its matching source revision.

Rollback is intended for an agent that has damaged its own active
image. It is not required for ordinary failed experiments that remain
recoverable in the live image.

* 10. Recovery

Recovery may begin through:

- =autolith --recovery=
- A fatal recovery exit status.
- An unexpected process failure detected by the launcher.

The launcher passes the recovery image whatever context is available.

A crash capsule may contain:

- Condition report.
- Backtrace.
- Exit status or signal.
- Recent conversation events.
- Recent tool calls.
- Current Git revision.
- Current generation.
- Mutation-journal position.
- Recently changed definitions.
- Paths to pending source changes.

The recovery image may:

1. Inspect the crash capsule.
2. Inspect the failed source and mutation history.
3. Ask the model to diagnose the failure.
4. Patch the source.
5. Load and test it in a disposable Lisp worker.
6. Build a repaired image.
7. Select the repaired image as active.
8. Fall back to a retained working generation when repair fails.

Recovery must remain possible without loading the damaged active core.

* 11. Dependencies and External Tools

Autolith should include or locate:

- A pinned SBCL runtime.
- The exact source corresponding to the pinned SBCL runtime.
- A local, pinned Quicklisp distribution or equivalent dependency set.
- Required Lisp source dependencies.
- Bubblewrap and a private =cl-exec-sandbox= helper on Linux.
- A documented set of useful external binaries.

Available Lisp systems and external commands are exposed to the agent
through a capability registry.

Ordinary filesystem, process, networking, and build operations should
prefer Common Lisp, ASDF, and UIOP.

On the initial Linux x86-64 target, setup may install a hash-verified pinned
SBCL binary beneath the data root when no compatible runtime is available. The
launcher and development commands use the recorded executable without changing
the user's system Lisp installation.

The matching SBCL source is a reference and reconstruction input for
implementation-aware inspection. Live implementation mutation changes only a
Lisp heap or a newly published worker core. Normal tools never overwrite the
pinned SBCL runtime executable, and the recovery image continues to boot with
its separately validated pristine runtime and core.

Autolith should not generate ad hoc Python or shell files unless a
non-Lisp dependency genuinely requires them.

* 12. Terminal Interface

The first interface is a restrained TUI optimized for dogfooding.

It should show:

- Conversation.
- Relevant persistent memory activity through tool calls.
- Streaming model output.
- Tool calls and results.
- Conditions and selected restarts.
- Current model.
- Current generation.
- Background logs on demand.

The interface should remain responsive during tool execution and model
streaming.

While a turn is active, Enter steers its next tool boundary and Tab queues a
later follow-up. Pending inputs are previewed in the live region, and an empty
Tab recalls the newest follow-up for editing. While idle, Tab submits like Enter.

=/later INPUT= persists INPUT with its current workspace and dispatches it after
the provider's relevant rate-limit reset. An exhausted longer window governs;
when the primary window is absent, a bounded five-hour estimate may be used.
Deferred state is private readable data, survives restart, and supports listing
and exact cancellation.

Ctrl-C with an empty input exits the interface. When the active conversation
has durable records, shutdown displays the exact command that resumes its
identifier. An empty conversation has no resume command.

The startup mark uses a green gradient during ordinary startup and a red
gradient when the recovery component starts the active process. Terminals
without indexed colors receive the corresponding solid bold basic color.

Each startup banner includes one tip selected from the canonical interactive
commands. The command token is colored separately from its explanation. Every
canonical command definition carries a non-empty tip, and omitting it is a
compile-time error. Declared aliases share their canonical command's tip.

Decorative animations, excessive transitions, and visual spectacle are
out of scope.

* 13. Security Status

Autolith is not a security sandbox.

It executes model-generated Lisp with the privileges of the user running it.
Ordinary =shell.run= commands require approval and use the workspace command
sandbox described above. The user can explicitly allow full command access for
the current session.

Command containment, process separation, omitted tools, checkpoints, and
recovery mechanisms are intended to reduce accidental permanent damage. The
application as a whole is not designed to contain malicious code.

The program must display this limitation clearly in its documentation
and initial setup.

* 14. Initial Milestone

The first dogfoodable version is complete when it can:

1. Authenticate through the Codex subscription provider.
2. Hold a streaming conversation in the TUI.
3. Save and reload the conversation from a =.sexp= file.
4. Execute =lisp.*= operations in named, heap-isolated SBCL REPL processes.
5. Inspect and redefine a live function through =self.*=.
6. Persist that definition as a private image commit and replay script.
7. Create a Linux checkpoint without stopping the active agent.
8. Recover from an intentionally broken agent loop.
9. Boot a retained generation through =autolith --recovery=.
10. Rebuild a working image from the tracked source and pinned
    dependencies.
